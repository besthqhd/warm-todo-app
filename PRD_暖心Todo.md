## 2. 项目背景

### 2.1 业务背景
自我管理类工具（待办、习惯、番茄钟）市场成熟，但用户**留存率低、弃用率高**。核心矛盾在于：工具越「专业」越「复杂」，用户初期热情消退后难以坚持。市面主流产品要么偏「任务管理」（冷冰冰的清单），要么偏「习惯打卡」（缺乏与日常事务的关联）。

### 2.2 用户痛点
- **启动门槛高**：面对空白清单不知从何写起，决策疲劳。
- **待办与习惯割裂**：完成了待办，习惯却没打卡；两者无法相互带动。
- **优先级混乱**：重要的事总被紧急但不重要的事挤掉。
- **缺乏陪伴感**：传统工具是「记录器」，不是「协作者」，用户遇到卡点无人引导。
- **复盘困难**：周/月回顾时看不到自己的规律与盲区。

### 2.3 市场机会
- **AI 原生交互**是差异点：用自然语言建待办、智能拆解任务、情感鼓励，把「坚持」变成有反馈的对话。
- **待办×习惯联动**形成闭环，提升长期留存。
- **温暖治愈视觉**降低使用焦虑，契合年轻用户（学生 / 职场新人）审美。

## 3. 产品目标

### 3.1 核心目标
1. **降低启动阻力**：AI 帮助用户把模糊意图（"我想备考"）变成结构化任务树。
2. **打通待办—习惯闭环**：完成待办引导打卡，完成习惯引导处理待办。
3. **可坚持**：通过情感鼓励 + 可视化复盘，提升 4 周留存。

### 3.2 成功指标（北极星 + 关键指标）

| 指标 | 定义 | 目标（上线 90 天） |
|------|------|------|
| **北极星**：周活跃留存 | 注册后第 7 日仍活跃用户占比 | **≥ 35%** |
| 待办创建转化率 | 新用户首日创建 ≥1 条待办占比 | ≥ 60% |
| 习惯联动率 | 完成待办后触发习惯打卡的占比 | ≥ 25% |
| AI 功能渗透率 | 月活中用过 AI 对话的占比 | ≥ 40% |
| 任务拆解采纳率 | AI 提议拆解被用户「接受/微调」占比 | ≥ 50% |

### 3.3 时间规划（建议）
- **M1（第 1–4 周）**：待办 + 习惯 + 四象限 + 基础导航（存量已实现）。
- **M2（第 5–8 周）**：AI 对话 / 自然语言建待办 / 智能习惯推荐 / 情感鼓励。
- **M3（第 9–12 周）**：长上下文记忆 + RAG + 交互式任务拆解 + 数据看板。

## 4. 用户画像

### 4.1 目标用户
| 画像 | 标签 | 典型特征 |
|------|------|----------|
| **学生党·小满** | 18–24 岁，大学生 | 备考 / 论文 / 社团并行，需要拆解大目标，易拖延 |
| **职场新人·阿哲** | 22–28 岁，初级白领 | 事务杂、怕遗漏，想建立运动 / 阅读习惯 |
| **自律进阶·林姐** | 28–35 岁 | 已有习惯体系，需要复盘与优先级管理 |

### 4.2 用户故事（核心）
- **作为**学生小满，**我希望**把"准备考研"说给 AI 听，**以便**它帮我拆成可执行的周计划。
- **作为**职场新人阿哲，**我希望**待办能标重要紧急，**以便**先处理真正要紧的事。
- **作为**用户，**我希望**完成待办后顺手打卡一个习惯，**以便**两个目标不脱节。
- **作为**用户，**我希望**卡住时 AI 给我鼓励而非冷冰冰提示，**以便**我愿意继续。

### 4.3 使用场景
- **晨间规划**：打开 App → 看今日待办 + 习惯 → 用 AI 把当天任务排好。
- **碎片记录**：想到一件事 → 自然语言一句话建待办（"明天下午 3 点交报告"）。
- **晚间复盘**：总览页看完成度与象限分布 → 决定明天重点。

## 5. 功能列表

> 优先级采用 **P0（必做）/ P1（重要）/ P2（增强）**；验收标准以「给定—当—则」（Given-When-Then）描述，供测试写用例。

### 5.1 待办管理

| 功能 | 用户价值 | 优先级 | 验收标准 |
|------|----------|--------|----------|
| **创建待办（自然语言）** | 一句话建任务，零表单负担 | P0 | 给定输入框，当用户输入"明天下午3点交报告"并发送，则生成待办：标题=交报告、时间=次日15:00、默认象限=重要不紧急 |
| **四象限分类** | 区分重要/紧急，避免瞎忙 | P0 | 当创建待办选择象限，则列表按 重要紧急/重要不紧急/不重要紧急/不重要不紧急 分组展示 |
| **多级子任务** | 大目标拆成可执行步骤 | P0 | 当某待办展开，则可无限层级添加子任务（名称/类型/象限），总览递归展开任务树 |
| **编辑 / 删除 / 完成** | 基本生命周期 | P0 | 当勾选完成，则触发"待办完成询问习惯"弹窗（见 6.3） |

### 5.2 习惯管理

| 功能 | 用户价值 | 优先级 | 验收标准 |
|------|----------|--------|----------|
| **新建习惯** | 建立长期行为 | P0 | 当新建习惯填名称+频率+分类，则进入"习惯展示"页对应分类 |
| **分类打卡** | 可视化坚持 | P0 | 当点击打卡，则当日标记完成、连续天数+1、展示环进度 |
| **智能习惯推荐（AI）** | 降低想"养什么习惯"的决策成本 | P1 | 当用户输入目标（如"想更健康"），则 AI 返回 2–3 个推荐习惯卡片，可一键添加 |
| **待办→习惯联动** | 闭环 | P0 | 当完成待办，则弹窗引导打卡关联习惯；当完成习惯，则引导处理待办（双向联动） |

### 5.3 AI 智能助手

| 功能 | 用户价值 | 优先级 | 验收标准 |
|------|----------|--------|----------|
| **AI 对话页** | 统一入口 | P0 | 当点击底部导航 AI，则打开 AI 对话页（界面保持在手机框内，不撑满全屏） |
| **自然语言建待办** | 语音/打字即建 | P0 | 见 5.1 验收标准 |
| **NL 查询** | 问"我这周完成了什么"即得答 | P1 | 当用户输入查询类语句，则基于本地数据返回结构化回答 |
| **情感鼓励** | 陪伴感 | P1 | 当检测到用户连续未完成或表达挫败，则返回温和鼓励文案（非说教） |
| **交互式任务拆解** | 大目标→任务树 | P1 | 当用户输入模糊大目标，则 AI 先提议拆解方案 → 询问确认 → 接受用户自己的思路整理后生成 |
| **长上下文记忆** | 越用越懂你 | P2 | 当多次对话后，AI 能引用历史偏好（如"你上次说备考"） |
| **RAG 检索** | 回答有据 | P2 | 当 AI 检索历史，则优先客户端 embedding 语义检索，无接口时降级本地词法 |

### 5.4 账户与设置

| 功能 | 用户价值 | 优先级 | 验收标准 |
|------|----------|--------|----------|
| 登录 / 注册 / 忘记密码 | 基础账户 | P0 | 三页流程闭环，注册成功 / 重置密码成功有反馈弹窗 |
| 我的页（设置入口） | 集中管理 | P0 | 当点击"我的"，则展示设置入口（含 AI 配置、编辑资料、欢迎语编辑） |
| 编辑资料 / 欢迎语编辑 | 个性化 | P1 | 当编辑欢迎语，则底部弹层编辑并实时保存 |
| AI 配置（Embedding 模型/地址） | 高级可配 | P2 | 当修改 Embedding 配置，则可重建索引并提示成功 |

### 5.5 视图与复盘

| 功能 | 用户价值 | 优先级 | 验收标准 |
|------|----------|--------|----------|
| 日历·月视图 / 周视图 | 时间维度规划 | P0 | 当切换周视图，则可点选日期查看当日任务 |
| 总览（统计+图表） | 复盘 | P0 | 当打开总览，则展示完成度、象限分布、任务树（可递归展开） |
| 每日规划（时间轴） | 当日编排 | P1 | 当打开每日规划，则按时间轴排列当日待办 |

### 6.2 交互流程（主链路）
```
启动 → 登录/注册（或游客）→ 首页
  ├─ 添加待办 → 选象限 → 存首页 →（完成）→ 弹窗联动习惯
  ├─ 新建习惯 → 分类打卡
  ├─ 总览 → 复盘
  └─ 底部导航[AI] → AI 对话 → 自然语言建待办 / 任务拆解 / 情感鼓励
```

### 6.3 边界情况（验收必覆盖）
- **AI 页不撑满**：点击导航 AI 后，界面须保持在 `360×798` 手机框内（`page-ai` 须位于 `.phone-frame` 内，DOM 标签须闭合）——**回归用例**。
- **无网络 / 无后端**：AI 功能降级为本地规则，界面不报错、不白屏。
- **空状态**：无待办 / 无习惯时，首页展示引导文案而非空白。
- **超大文本**：待办标题超长时换行不溢出卡片边界（参考"我的页设置行"历史 bug）。
- **时区/跨天**：23:59 创建的"明天"任务须正确归属次日。
- **多级子任务**：无限层级时不栈溢出、总览可递归展开。

## 7. 数据需求

### 7.1 埋点需求（事件表）

| 事件名 | 触发时机 | 关键属性 |
|--------|----------|----------|
| `todo_create` | 创建待办 | source(nl/ui)、quadrant、has_subtask |
| `todo_complete` | 完成待办 | from_linkage(bool) |
| `habit_create` / `habit_check` | 建/打卡习惯 | category、streak |
| `ai_session_start` / `ai_message_send` | AI 对话 | intent(nl_create/decompose/query/encourage)、model |
| `ai_decompose_accept` | 采纳拆解 | depth、node_count |

### 7.2 数据看板（建议指标）
- **活跃**：DAU / WAU / 7 日留存。
- **转化漏斗**：启动 → 创建首条待办 → 完成首条 → 建立习惯。
- **AI 价值**：AI 渗透、拆解采纳率、NL 建待办占比。
- **闭环健康度**：联动触发率、习惯连续打卡分布。

## 8. 风险与依赖

### 8.1 技术风险
| 风险 | 影响 | 应对方案 |
|------|------|----------|
| **AI 接口不稳定 / 超时** | 对话卡死、用户体验崩塌 | 前端设超时与重试；无响应时降级本地规则并提示"稍后再试" |
| **DOM 结构错乱导致页面撑满** | AI 页渲染异常（**已发生**） | 标签栈 CI 校验（div 配对）+ 每次发布前无头渲染截图回归 |
| **RAG embedding 接口缺失** | 检索失效 | 客户端语义检索为主，无接口自动降级本地词法兜底 |
| **长上下文超 token** | 成本/截断 | 用户画像 + 历史摘要注入，按窗口裁剪 |

### 8.2 外部依赖
- **LLM 服务**：需可配置 Base URL / Key（设置页）；默认走 `server/` 代理。
- **Embedding 服务**：可选，用于 RAG；缺省降级。
- **GitHub Pages**：静态托管与在线预览。

### 8.3 非技术风险
- **AI 建议准确性**：拆解可能不合用户实际 → 采用"先提议、再确认"交互，用户始终有最终编辑权。
- **隐私**：聊天/画像本地持久化 → 明确告知并在设置提供清除入口。
