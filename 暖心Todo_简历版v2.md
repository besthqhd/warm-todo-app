## 一、简历正文（项目经历栏，建议 5-6 行，术语已稀释进业务）

**暖心Todo ｜ AI 情感化待办 / 习惯管理 App（0→1 个人产品；前端原型已公网部署 GitHub Pages，服务端智能体为本地可运行版本）**

- **问题与定位**：审计滴答清单 / Notion Todo / 微软 To Do 等发现，主流待办普遍聚焦"效率管理"，缺失"大任务的情绪拆解"与"习惯坚持的情感反馈"；社区观察（即刻 / 小红书 / Reddit r/productivity）印证"知道该做却迈不出第一步"是共性痛点，据此将产品定义为"AI 拆解教练 + 记忆 + 检索"，而非更快的清单。
- **架构决策（3 取舍）**：① 单接口 + 9 工具场景自研 ReAct 循环（~10KB 零依赖），不引入 LangChain；② 离线无 Embedding 时改双模降级（语义优先 + 词法兜底），无 key 仍可检索；③ 拆解从直调改为智能体通道（开放查询工具子集），模型自主"先查再拆"、步骤可视化。
- **AI 流程设计**：设计交互式拆解——模型先提议、再询问、支持"采纳 / 自备 / 换拆法"三态确认，将不可控生成收敛为可确认可编辑流程，平衡"模型自主"与"用户掌控感"。
- **问题排查与修复**：6 轮反馈驱动迭代，修复 4 个 bad case——多级任务展开失效（根因 addTodo ID 毫秒碰撞，随机后缀后 **0% → 100%**）、导航 AI tab 与设置入口缺失（**2 → 0**）、子任务死按钮（**1 → 0**），每轮功能测试回归。
- **验证与协作**：建立 **30+ 功能断言 100% 通过**（ID 唯一性 / 检索召回 / 导航一致性，tests/assertions.js 实测）；设计 6 维度评测及与通用模型对照打分方法（已设计、待执行）；独立完成前端原型（公网部署）+ 服务端智能体（Node + ReAct），主导安全设计（密钥本地化、不入库、服务端仅从请求体取 key）。

## 二、项目扩展叙事（面试口述，按"问题 → 决策 → 效果 → 迭代"）

### 1. 问题定义（why，诚实版）
- **竞品功能审计**：主流待办产品（滴答清单 / Notion Todo / 微软 To Do / Things 等）能力高度同质，普遍聚焦效率，无一款处理"情绪拆解"与"情感陪伴"——这是市场空白。
- **社区观察**：即刻"自律"话题、小红书"拖延症"笔记、Reddit r/productivity 高频讨论均指向"知道该做什么但迈不出第一步"——印证方向，而非凭空假设。
- **判断**：差异化机会在"会拆解 + 记得住你 + 给正向反馈"，而非"更快的清单"。
- **诚实标注**：个人项目，未做正式问卷 / 访谈；需求验证方式为 **竞品功能审计 + 社区观察 + 身边 3-5 人试用反馈**，非大样本统计。

### 2. 架构决策（3 个取舍，否定 X → 选 Y → 效果 Z）
| 决策 | 遇到的问题 | 否决的方案 | 选择的方案 | 效果 |
|---|---|---|---|---|
| 是否用框架 | 单 OpenAI 兼容接口 + 9 工具，场景简单 | 引入 LangChain（重依赖、抽象层） | 自研 ReAct 智能体循环（~10KB 零依赖） | 维护成本低、逻辑可控、易读 |
| 检索容错 | Embedding 接口缺失 / 调用失败时应仍能用 | 纯语义检索（无 key 即不可用） | 双模降级：向量语义优先 + 词法兜底 | 无 API key 时 RAG 仍可用 |
| 拆解实现 | 单次模型调用无法查询用户已有数据 | 直调 LLM 出拆解方案 | 改智能体通道（开放查询类工具子集） | 模型自主决定"先查再拆"，步骤可视化 |

### 3. 功能背后的 AI 能力（面试口述、由简入繁，不进简历正文）
- **提示层**：主系统提示定义暖心助手角色与行为约束（先查询后创建）；拆解场景用独立子提示限定输出格式（结构化 JSON 任务树）；动态拼装【长期记忆 + 检索片段】。
- **上下文层**：长上下文记忆（用户画像 / 情绪 / 偏好持久化，跨会话不"失忆"）；检索增强（RAG）语料来自 5 类（待办含多级路径 / 习惯 / 打卡 / 记忆 / 聊天），语义优先、词法兜底；语料哈希缓存避免重复计算。
- **循环层**：智能体 ReAct 循环（最多 6 步 / 9 工具），模型依据返回自主决策调用顺序；外层三态确认（Workflow）收敛不可控生成，保证可控可预期。

### 4. 量化指标与评测（诚实分层）
- **真实技术指标（可现场演示）**：30+ 条功能断言 100% 通过（tests/assertions.js 自动化实测，覆盖数据层 ID 唯一性 / AI 层检索召回命中 / 交互层导航一致性）；多级任务展开成功率修复前后 **0% → 100%**；导航 / 设置缺失项 **2 → 0**；交互死节点 **1 → 0**。
- **结构化输出容错评测（真实测量）**：针对模型任务树 JSON 输出不稳定，搭建 20 条畸形输出评测集（截断 / 字段漂移 / 格式变异），实测解析准确率 85%；定位未闭合代码块漏检 + 空数组误判两类失效，修复后准确率 **95%**、误报漏报由 **3 例降至 1 例**（tests/eval_json_parse.js 可现场复跑）。
- **评测体系（已设计，建议面试前执行并填数）**：
  - 方法：准备 10-15 个典型大任务拆解请求，同时发给暖心Todo AI 与通用模型（如 ChatGPT），人工按 6 维度（拆解颗粒度 / 层级合理性 / 与用户意图匹配 / 可执行性 / 是否结合历史 / 语气暖度）各打 1-5 分
  - 产出：综合得分、维度覆盖率、与基线的相对提升 —— 执行后填入简历
  - 现状标注：个人项目阶段聚焦功能正确性与工程可行性验证，评测集已就绪，不编造用户规模数据
- **原则**：业务指标只写"已设计评测方法"，真实得分执行后补，绝不前置编造。

### 5. 数据 / 反馈驱动迭代记录（每轮都有触发来源）
- 第 2 轮触发（反馈）：用户反馈"总览点不开一级任务" → 排查定位 addTodo ID 冲突根因 → 修复，展开率 0→100%。
- 第 3 轮触发（反馈）：用户要求"子任务也要类型和象限、且不限层级" → 从扁平子任务重构为递归节点树（无限层级）。
- 第 4 轮触发（决策）：判断"拆解不应写死分支" → 改造为智能体自主决策通道（模型自决先查还是先拆）。
- 第 5 轮触发（审计）：AI 工程三层审计发现上下文层空白 → 落地 RAG 检索增强。
- 第 6 轮触发（反馈）：用户截图指出 AI 页尺寸 / 导航不一致 → 补底部导航、统一背景与 padding。

## 三、面试 STAR 话术要点

- **"为什么用 RAG 而不是直接让模型回答？"**
  → 长上下文记忆解决"记住用户"，检索增强解决"基于真实历史回答"。用户问"我上周季度汇报后来怎样了"，没检索模型只能编；有检索能从历史待办捞到真实进度。取舍上先做低成本的记忆、再上较重的检索（依赖关系 + 投入产出比）。

- **"Agent 和 Workflow 怎么区分？"**
  → 内核是智能体（模型自主决定调哪个工具、什么顺序）；外壳是 Workflow（三态确认由产品写死，保证可控）。主流做法：不完全放权、也不完全写死。

- **"遇到最难的技术问题？怎么排查？"**
  → 多级任务展不开。现象：点一级任务不出子任务。排查：先怀疑渲染逻辑（改 CSS 切换无效），再 dump 数据发现父子链断。根因：addTodo 用 Date.now() 做 ID，批量插入同毫秒碰撞。修复：随机后缀唯一 ID。验证：写功能测试断言"0 重复 ID + 层级树正确"，此后每次改动都回归。

- **"如果用户滥用 API 怎么办？"**
  → 密钥仅存本地、不进仓库（已核查 public repo 无真实 key）；服务端智能体当前为本地可运行版本，若后续公网部署为代理需补限流 + 鉴权。体现成本与安全意识。

- **"你做过用户调研吗？"**（诚实答法，别编）
  → 个人项目未做正式问卷 / 访谈；需求验证靠竞品功能审计（主流待办产品能力同质、缺情感层）+ 社区观察（即刻 / 小红书 / Reddit 高频痛点一致）+ 身边 3-5 人试用反馈。上线后会用已设计的 6 维度评测体系做量化验证。

## ⚠️ 简历避坑清单（v2 强化版）

1. **不写"我用 React/Node 从零开发了 App"**——工程师视角，易在追问下露怯。写成"主导产品设计、定义 AI 能力、与工程协作落地"。
2. **AI 术语稀释进业务叙事**——绝不单开一段"Prompt/Context/Loop Engineering"罗列；只在讲具体决策时自然带出（如"用长上下文记忆让 AI 跨会话记得住用户"）。
3. **未做正式用户调研 → 一律写"竞品审计 + 社区观察"**——不编问卷样本量、不编百分比（警惕"63.64%"这类精确假数）。
4. **所有数字必须可证**：0→100% 展开率、2→0 缺失项、30+ 断言通过（tests/assertions.js 实测），都是真实功能事实；评测打分**执行后填**，不前置编造。
5. **面试可现场演示验证**：带着 tests/assertions.js 与 eval_json_parse.js 当场跑，所有数字（断言 100% / 解析准确率 95%）都是真测的，比嘴说更有说服力。
6. **MVP 阶段思维**：6 轮迭代按 v1-v6 讲清每阶段目标 + 验证方式，体现"0→1 产品节奏感"。
