# 暖心Todo · Agent 化改造方案

> 目标：把当前「静态 App + 外挂 LLM 调用」升级为「能感知、会规划、可调工具、有记忆」的真·AI Agent。
> 配套文档：`PRD_暖心Todo.md`（产品需求）、`index.html`（现状实现）。

---

## 0. 为什么需要改造（现状差距）

当前实现的 4 个 AI 能力里，只有「自然语言建待办」「自然语言查询」真正调用了 LLM，且都是**单次请求 + 硬编码解析**。本质是「伪 Agent」，缺口如下：

| # | 缺口 | 现状 | 理想 Agent |
|---|------|------|-----------|
| 1 | 无工具调用 | AI 只在聊天框说话，3 个写死按钮；模型不能自己决定"该建待办/该查数据" | 模型按意图**自主选择**并调用工具 |
| 2 | 无自主规划 | 一次对话 = 一次 API 调用，无多步推理 | ReAct 循环：思考→调工具→看结果→再思考 |
| 3 | 无服务端代理 | API Key 在浏览器（不安全、CORS、易刷爆） | Key 只在服务端，浏览器零密钥 |
| 4 | 无长期记忆 | 仅 localStorage，换设备即失忆；规则引擎与 LLM 互不打通 | 结构化画像 + 跨会话记忆 |

## 2. 核心组件

### 2.1 Agent 服务端（新增）
- **语言**：Node.js + Fastify（推荐，与前端同源 JS，运维简单）；备选 Python + FastAPI。
- **职责**：
  - 持有 LLM API Key（环境变量 `LLM_API_KEY`，**绝不下发浏览器**）
  - 实现 ReAct / Function-Calling 循环
  - 暴露 `POST /api/agent/chat` 给前端
  - 鉴权 + 限流 + CORS 白名单

### 2.2 工具注册表（Tool Registry）
每个工具用 OpenAI function-calling 的 JSON Schema 描述，模型可自主调用：

| 工具 | 入参 | 作用 |
|------|------|------|
| `create_todo` | name, type, quadrant, subtasks[] | 新建待办（替代现有写死解析） |
| `list_todos` | filter(状态/类型/日期) | 列出待办 |
| `complete_todo` | id | 标记完成 |
| `create_habit` | name, freq | 新建习惯 |
| `checkin_habit` | id | 习惯打卡 |
| `query_stats` | range(本周/本月) | 返回完成率、拖延次数等 |
| `recommend_today` | — | 基于记忆推荐今日待办/习惯 |
| `encourage` | — | 生成情感鼓励文案 |
| `get_current_time` | — | 获取当前时间（规划用） |

> 现有的 `recommendHabits` / `encourageMe` 规则引擎**不删除**，而是封装成工具，让 Agent 能自主决定何时调用——实现"规则 + 大模型"混合推理。

### 2.3 记忆层升级
- **短期**：当前会话上下文（服务端内存，随对话窗口滑动）
- **长期（MVP）**：localStorage（客户端，UI 真源）+ 结构化用户画像（如"偏好晨间习惯""常拖延类型"）
- **未来（Phase 4）**：可选向量库做语义回忆（如记住"用户上周说想健身"）

### 2.4 主动能力（Proactive）
- 后端定时器 / 前端 Service Worker：到点检查「逾期未办」「低完成率」→ 推送浏览器通知（`Notification` API）
- 把 App 从"你问我答"变成"它会主动关心你"

## 3. 两种落地路线（关键决策点）

### 路线 A：薄代理 + 客户端执行工具（MVP 推荐）
- 服务端**只做 LLM 编排 + 函数调用循环**，工具实际在浏览器执行
- 数据真源仍是 localStorage，**改动最小**，现有 105KB App 几乎不动
- 流程：
  1. 浏览器发消息 → 服务端带 `tools` 调 LLM
  2. LLM 返回 `tool_call` → 服务端回传浏览器
  3. 浏览器执行工具（改 localStorage）→ 把 observation 发回服务端
  4. 服务端继续循环，直到 LLM 给出最终回复

### 路线 B：全后端 + 数据库（完整但重）
- 服务端持有数据（SQLite / Postgres），工具在服务端执行，UI 同步
- 多设备、真持久化，但要重写整个数据层
- 适合 Phase 5 规模化

**建议**：路线 A 起步（快速拿到"真 Agent"体验），B 作为 Phase 5 演进。

## 4. 对现有代码的改动

| 现有实现 | Agent 化后 |
|---------|------------|
| `callLLM()` 浏览器直连 LLM | → `fetch('/api/agent/chat')`（经服务端） |
| `handleNLCreateTodo` 写死 JSON 解析 | → Agent 调 `create_todo` 工具，由模型结构化 |
| `recommendHabits` / `encourageMe` 规则独立 | → 封装为 Agent 可自主调用的工具 |
| 设置页存 Key 到 localStorage | → Key 移入服务端环境变量 |
| 3 个写死快捷按钮 | → 模型自己选工具；按钮降为"建议快捷指令" |

**新增**：
- `server/` 目录（后端：ReAct 循环 + 工具注册 + 记忆 + 调度）
- 前端「工具执行桥」：收到 `tool_call` 后在浏览器落地
- 部署配置（见第 6 节）

## 5. 分阶段实施

| 阶段 | 内容 | 交付标志 |
|------|------|---------|
| **Phase 1（MVP）** | 后端代理 + Function Calling，模型能自主建待办 / 查数据 | 对话中模型自己调 `create_todo` 成功 |
| **Phase 2** | ReAct 多步：查统计→决策→建单→鼓励 一条龙 | 一句"帮我安排这周"自动完成多步 |
| **Phase 3** | 主动调度 + 浏览器通知 | 逾期/低完成率时主动推送 |
| **Phase 4** | 长期记忆 / 个性化画像 | 跨会话记住用户偏好 |
| **Phase 5** | 多设备同步（路线 B 数据层） | 换设备数据不丢 |

## 6. 部署变化（重要）

> ⚠️ **GitHub Pages 不能托管后端**（它只服务静态文件）。需调整：

| 方案 | 说明 |
|------|------|
| **A. 前端留 Pages + 后端挂外部** | 前端仍在 `besthqhd.github.io/warm-todo-app`；后端挂 Railway / Render / Fly.io / Vercel Functions |
| **B. 整体迁 Vercel（推荐）** | 前端 + `/api` serverless 同仓一套搞定，零额外运维，自定义域名方便 |

- Key 进服务端环境变量，前端域名配 CORS 白名单
- 若走方案 B，GitHub 仓库可继续作为源码仓，Vercel 连仓库自动部署

---

## 7. 安全与边界

- API Key 永不出服务端
- 工具执行需会话鉴权（防越权改他人数据）
- 限流防刷（按会话/IP）
- 模型返回的 `tool_call` 参数做 JSON Schema 校验，防注入 / 越权工具调用
- 用户可随时在设置页关闭"主动通知"

---

## 8. 风险与权衡

| 风险 | 说明 | 缓解 |
|------|------|------|
| 运维成本 | 后端需持续运行 | 用 serverless（Vercel Functions）免运维 |
| 冷启动延迟 | serverless 首次慢 | 预热 / 常驻小实例 |
| Token 成本 | 多步循环消耗上升 | 工具结果摘要化、限制循环步数 |
| 体验回退 | 网络/后端挂了 AI 不可用 | 保留规则引擎兜底（推荐/鼓励不依赖 LLM） |

## 9. 下一步建议

1. **确认两个决策**：① 后端语言（Node 推荐）/ ② 落地路线（A 薄代理 推荐） / ③ 部署（Vercel 推荐）

3. 先不推翻现有 App，以"可切换"方式接入（设置页加开关：本地直连 / Agent 服务端）
